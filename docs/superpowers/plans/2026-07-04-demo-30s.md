# 30-Second Demo Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** One-command demo (`make demo`) that boots Suitest + a bundled "Brewly" demo app with a seeded, runnable suite, plus a launch GIF embedded in the README.

**Architecture:** Brewly is a standalone FastAPI fixture app in `examples/demo-app/` added to docker compose under a new `demo` profile. A one-shot `demo-seed` service (reusing the idempotent `Seeder` pattern from `suitest_db.seed`) creates a Demo workspace/project and imports a committed suite fixture whose steps execute against the live demo-app container (BE_REST via `api-mcp`, FE_WEB via `playwright-mcp`). The GIF is recorded from the real running stack with a Playwright recording script, converted with ffmpeg, committed under `assets/`, and embedded at the top of `README.md`.

**Tech Stack:** FastAPI, SQLAlchemy async (existing seed infra), Docker Compose profiles, Playwright (recording), ffmpeg.

**Spec:** `docs/superpowers/specs/2026-07-04-demo-30s-design.md`

**Known constraints discovered during research:**
- ZERO tier requires every step to carry explicit `code` (`STEPS_REQUIRE_CODE_IN_ZERO_LLM`) â€” the fixture suite must ship `code` on every step.
- Step routing: `target_kind=BE_REST` + `mcp_provider="api-mcp"` (code = HTTP request JSON); `target_kind=FE_WEB` + `mcp_provider="playwright-mcp"` (code = Playwright JS).
- MCP providers are per-workspace rows â€” demo seed must register `api-mcp` and `playwright-mcp` for the Demo workspace.
- Auth is cookie-based (`POST /auth/cookie/login`); no signup â€” seed must create the demo user.
- If `playwright-mcp` proves unavailable inside the dockerized runner during Task 5 smoke, drop the FE_WEB case from `suite.json` (API-only suite still lands green; artifact = logs).
