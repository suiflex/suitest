# 30-Second Demo — Design

**Date:** 2026-07-04
**Status:** Approved
**Goal:** Make Suitest demo-able in 30 seconds: a launch GIF in the README showing
PRD → generated suite → green run, plus a one-command replay path (`make demo`)
that works with or without an LLM key.

## Motivation

Launch distribution depends on a demo people can watch in 30 seconds and replay
in one command. The wow moment is the CLOUD-tier flow (PRD → AI-generated suite
→ green run), but replay must never dead-end for users without an LLM key
(ZERO-first rule).

## Components

### 1. Demo app — `examples/demo-app/` ("Brewly")

Small coffee-shop order app used as the testing target.

- FastAPI, single service, SQLite/in-memory. No auth. < ~300 lines.
- REST API: menu list, order CRUD, checkout with stock validation.
- Minimal web UI (static HTML + fetch) served by the same app.
- `openapi.json` exported from FastAPI.
- `PRD.md` (~1 page, English): menu, ordering, stock, discount requirements.
  This is the generation input shown in the demo.
- `Dockerfile` for the compose service.
- The demo app is a fixture, not a product. It gets no roadmap entries.

### 2. Compose profile `demo`

- New service `demo-app` (port 8089), `profiles: ["demo"]`.
- New one-shot service `demo-seed` (same pattern as `migrate`/`minio-init`):
  runs a Python seed script that creates workspace "Demo" + project "Brewly",
  registers targets (demo-app API + web), stores `PRD.md`, and imports a
  **pre-generated suite** — a real AI-generation output committed as a JSON
  fixture (`examples/demo-app/suite.json`), following the `examples/*/case.json`
  pattern. Seeding is idempotent.
- Core services join the `demo` profile.
- `make demo` → `docker compose --profile demo up` → prints web URL + demo
  credentials when ready.

### 3. Replay paths

- **ZERO (no LLM key):** open dashboard → Brewly suite already present → Run →
  green + artifacts. Deterministic, ~30s. UI hint on the suite: "Generated by
  AI from PRD.md — add an LLM key to generate your own."
- **CLOUD/LOCAL (key configured):** Settings → LLM → generate from `PRD.md` →
  new suite appears → Run. This is the flow recorded for the GIF.

### 4. Launch GIF

Shot list (~30s total), recorded against the real stack:

1. 0–4s — terminal: `make demo`, compose up (sped up).
2. 4–8s — browser: dashboard, select `PRD.md`.
3. 8–18s — test cases appearing (violet AI badges). The wow moment.
4. 18–26s — Run → steps execute → green one by one.
5. 26–30s — dashboard: 100% pass, artifact screenshot open. Overlay tagline:
   "PRD → test suite → green. Open source. Your LLM."

Deliverables: MP4 + GIF under `assets/`, embedded at the top of `README.md`
(below the tagline, above Install). If real-LLM recording is not possible at
build time, record the ZERO replay flow (pre-generated suite → run → green) and
label it honestly; swap in the CLOUD recording later.

## Error handling

- `demo-seed` must be idempotent (safe on repeated `up`).
- Demo run must be deterministic: demo-app has fixed seed data; suite steps
  assert against it.
- If demo-app is unhealthy, seed fails loudly with a clear message.

## Testing

- Unit: seed script (creates entities, idempotent).
- CI smoke (optional, follow-up): `docker compose --profile demo up` →
  healthcheck demo-app + API → run the seeded suite headless → expect green.

## Out of scope

- Any platform feature changes. Friction found while recording becomes separate
  ROADMAP items.
- Marketing copy beyond the GIF overlay tagline.
